# draw-bitch — рисовалка-угадайка для игры с друзьями

Дата: 2026-06-29

## Суть

Реалтайм-игра в духе Skribbl. Хост создаёт лобби (лимит секунд на ход, число ходов),
кидает ссылку друзьям. Игроки заходят, вводят имя. Хост жмёт «Старт». Игроки по очереди
рисуют загаданное слово, остальные угадывают в чате; кто быстрее — тем больше очков.
После N ходов показывается итоговая таблица, лобби возвращается в исходное состояние
ожидания (хост снова может нажать «Старт»).

Один процесс-бэкенд держит всё состояние **в памяти**; браузер общается с ним по WebSocket.

## Зафиксированные решения

- **Слова:** встроенный список **русских** слов; рисующему предлагается **3 случайных**,
  он выбирает одно.
- **Очки:** угадавшие получают очки **по скорости** (быстрее → больше) + **рисующий**
  получает бонус, пропорциональный числу угадавших.
- **Панель рисования:** минимум — ~8 цветов, 3 размера кисти, ластик, очистка холста.
  Без undo и заливки.
- **«Раунд» = один ход рисования.** Число, которое задаёт хост — это количество ходов
  (рисующих) за игру. Очерёдность циклически идёт по игрокам.
- **Состояние:** в памяти, эфемерно. Перезапуск/деплой обрывает активные игры — для
  вечеринки приемлемо. Redis не используется.
- **Обрыв связи:** игрок помечается оффлайн, но остаётся в списке со своими очками;
  по той же ссылке (токен в localStorage) авто-переподключается. Если отвалился рисующий —
  ход досрочно завершается.
- **Хост:** первый вошедший. Жмёт «Старт», задаёт настройки. При уходе роль переходит
  следующему игроку.

## Архитектура

Повторяет проект **flatinfo** (FastAPI + uv backend, Next.js standalone frontend, образы
в GHCR, авто-деплой по cron, общий nginx с TLS).

- `backend/` — FastAPI, Python 3.13, uv. Тулинг: ruff, mypy, pytest (как в flatinfo).
  Состояние игры — в памяти процесса. Redis **не** используется (в отличие от flatinfo).
- `frontend/` — Next.js 15 (output: standalone), React 19, TypeScript. Страница лобби —
  **клиентский** компонент (canvas + WebSocket), не SSR. Главная — простая форма.
- Инфра — как в flatinfo: Dockerfile'ы, `docker-compose.yml` (локальная сборка) +
  `docker-compose.prod.yml` (образы из GHCR), `.github/workflows/ci.yml`,
  `deploy/auto-update.sh`, `deploy/nginx/*.conf`, `Makefile`, `CLAUDE.md`.

### Машина состояний лобби

`WAITING` → `CHOOSING` (рисующий выбирает 1 из 3 слов) → `DRAWING` (рисует, идёт таймер,
остальные угадывают) → `TURN_END` (показали слово, начислили очки, короткая пауза) →
следующий ход (`CHOOSING` для следующего по очереди)… после N ходов → `GAME_END`
(итоговая таблица, пауза) → `WAITING`.

### Бэкенд: компоненты

- `app/config.py` — pydantic-settings (CORS_ORIGINS, лог-уровень, таймауты, размеры пауз).
- `app/game/lobby.py` — класс `Lobby`: чистая логика игры без I/O (настройки, игроки,
  очерёдность, текущий рисующий/слово, очки, множество угадавших, накопленные штрихи,
  чат). Методы переходов состояний возвращают события для рассылки. **Покрывается тестами.**
- `app/game/manager.py` — `LobbyManager`: словарь `Lobby` по id, создание/поиск/уборка
  пустых лобби.
- `app/game/words.py` — встроенный список русских слов + выбор 3 случайных.
- `app/game/scoring.py` — чистые функции расчёта очков (угадавший по доле времени,
  бонус рисующему). **Покрывается тестами.**
- `app/ws.py` — WebSocket-эндпоинт `/ws/{lobby_id}`: приём/валидация сообщений,
  управление соединениями, рассылка, реконнект по `player_id`, таймер хода (asyncio).
- `app/api.py` — REST: `POST /api/lobby` (создать, вернуть id+настройки),
  `GET /api/lobby/{id}` (существует ли), `GET /api/health`.
- `app/schemas.py` — Pydantic-модели входящих/исходящих WS-сообщений и REST.
- `app/main.py` — сборка FastAPI + CORS + роутеры.

### Протокол WebSocket (JSON)

Клиент → сервер: `join {name, player_id?}`, `start_game`, `choose_word {word}`,
`draw {segment}` (точки/линия с цветом и толщиной), `clear`, `chat {text}` (он же догадка),
`update_settings {round_seconds, total_turns}` (только хост, в WAITING).

Сервер → клиент: `joined {player_id, you}`, `state {phase, players, settings, scores, host_id,
turn, drawer_id, word_mask, time_left, canvas}`, `players`, `turn_start {drawer_id, word_mask,
duration}`, `word_choices {words}` (только рисующему), `draw {segment}`, `clear`,
`chat {player_id, text}`, `correct_guess {player_id, order}` (текст догадки скрыт),
`turn_end {word, scores}`, `game_end {scores}`, `error {message}`.

Полный снимок `state` шлётся при входе/реконнекте, чтобы поздно вошедший / вернувшийся
увидел текущую фазу, очки и перерисовал холст из накопленных штрихов.

### Угадывание

Нормализация: нижний регистр, trim, ё→е. Точное совпадение со словом → угадал. Сообщение
угадавшего не показывается остальным как текст (чтобы не палить слово) — вместо него
системное «угадал!». Рисующий и уже угадавшие могут переписываться отдельной системной
лентой (MVP: их сообщения видят только такие же «знающие»). Слово не раскрывается до
`TURN_END`. Подсказка для угадывающих — маска из подчёркиваний по числу букв.

### Очки

- Угадавший: `points = round(base * time_left / duration)` с минимальным порогом; раньше
  угадал → больше. (`base` — настраиваемая константа, напр. 100, плюс небольшой бонус за
  место в очереди угадывания.)
- Рисующий: бонус `= round(drawer_base * guessed / max(1, total_guessers))` — тем больше,
  чем больше людей угадали его рисунок.

### Фронтенд: компоненты

- `app/page.tsx` — главная: форма создания лобби (секунд на ход, число ходов) →
  `POST /api/lobby` → редирект в `/lobby/[id]`; и поле «вставь ссылку, чтобы войти».
- `app/lobby/[id]/page.tsx` — клиентская комната. Ввод имени (если нет в localStorage),
  открытие WS. Рендер по фазе: зал ожидания или игровое поле.
- `app/lobby/[id]/WaitingRoom.tsx` — список игроков, настройки (редактирует хост),
  кнопка «Старт» (у хоста), ссылка-приглашение с кнопкой «копировать».
- `app/lobby/[id]/GameBoard.tsx` — раскладка игры: `Canvas`, `Toolbar` (у рисующего),
  `Chat`, `Scoreboard`, `Timer`, маска/выбор слова.
- `app/lobby/[id]/Canvas.tsx` — HTML5 canvas; рисующий ловит pointer-события → шлёт
  сегменты; остальные применяют входящие `draw`/`clear`; перерисовка из снапшота при входе.
- `lib/useGameSocket.ts` — хук WS: соединение, реконнект с backoff, очередь исходящих,
  диспетч входящих сообщений в состояние.
- `lib/types.ts`, `lib/api.ts` — типы сообщений и REST-вызовы.

### Связь браузер ↔ бэкенд

Браузеру нужен прямой доступ к WS бэкенда (SSR-проксирование WS ненадёжно). Базовый URL —
`NEXT_PUBLIC_BACKEND_URL` (по умолчанию пусто = тот же origin). Прод: nginx роутит
`/api` и `/ws` на `127.0.0.1:8000`, `/` на фронтенд; бэкенд остаётся только на loopback.
Дев: `NEXT_PUBLIC_BACKEND_URL=http://localhost:8000`. CORS бэкенда разрешает домен/локалхост.

## Обработка ошибок и пограничные случаи

- Лобби не найдено (`GET`/WS) → ошибка, фронт показывает «лобби не существует».
- Старт игры при <2 игроках → запрет с сообщением.
- Рисующий отвалился во время `DRAWING`/`CHOOSING` → ход завершается досрочно.
- Все угадали раньше времени → ход завершается сразу.
- Никто не угадал → слово показывается, очков угадавшим нет, рисующему бонуса нет.
- Хост ушёл → роль следующему; если лобби опустело — удаляется по таймауту.
- Невалидное/неуместное по фазе сообщение → игнор + `error` отправителю.

## Тестирование

- Бэкенд: pytest на чистую логику — `Lobby` (переходы фаз, очерёдность, угадывание,
  завершение), `scoring` (формулы), `words` (выбор 3), `manager` (создание/уборка).
  Цель покрытия как в flatinfo (≥85% по логике; ws/main можно исключить из coverage).
- Фронтенд: typecheck + build в CI (как в flatinfo).

## Деплой и CI/CD

- `.github/workflows/ci.yml`: job `backend` (ruff+mypy+pytest), job `frontend`
  (typecheck+build), job `build-push` (на master: матрица backend/frontend → образы
  `ghcr.io/<owner>/drawbitch-backend:latest|sha`, `…-frontend:…`, кэш слоёв gha).
- `docker-compose.prod.yml` тянет образы из GHCR; `deploy/auto-update.sh` по cron делает
  `pull` и пересоздаёт контейнеры при изменении образов.
- `deploy/nginx/draw-bitch.conf`: TLS (Let's Encrypt webroot), `location /` → фронтенд,
  `location /api` и `location /ws` → бэкенд (с заголовками Upgrade/Connection для WS).

## Вне рамок MVP (YAGNI)

Аккаунты, БД, постоянное хранилище, приватные/публичные комнаты, кастомные словари,
undo/заливка, реакции, голосовой чат, расширенная мобильная вёрстка, антифрод/модерация.
